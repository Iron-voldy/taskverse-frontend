## Overview

TaskVerse is an AI-powered productivity platform where the UI itself is generated by AI based on what you ask. Instead of navigating a fixed interface, you describe your goal ("I need to track my workouts this week") and the app renders a custom widget layout — kanban boards, habit grids, focus timers, budget trackers, calendars — all composed on the fly using Gemini AI.

The frontend is a Next.js 16 App Router application with server-side rendering, streaming AI responses via SSE, real-time optimistic updates, and a gamification system (XP, levels, streaks).

## Features

### Dashboard
The dashboard is the home screen after login. It displays:

- **XP progress bar** — shows current level, XP earned, and XP needed for next level with an animated fill bar
- **Stats row** — total tasks, completed today, current streak (days), and active lists
- **Level-up banner** — appears when the user crosses an XP threshold
- **Task Calendar** — a mini monthly calendar where days with tasks show colored dots. Clicking a day filters tasks
- **Upcoming section** — the next 5 tasks due soonest, displayed in card variant (not row) to fit the 340px sidebar

### Task Manager
The `/app/tasks` page is the main task workspace:

- Filter by status (`todo`, `in_progress`, `done`, `archived`), priority (`low`, `medium`, `high`, `urgent`), and list
- Animated list using Framer Motion `AnimatePresence`
- Optimistic updates — UI reflects changes instantly, rolls back on API error
- AI-powered **task suggestion** — when creating a task, the `suggest` endpoint returns pre-filled priority, due date, tags, and estimated time based on the title
- **Ghost autocomplete** — as you type a task title, AI completes the sentence in grey text (press Tab to accept)
- Inline complete (awards XP + confetti), edit (modal), delete (confirm modal)

### GenUI Canvas
The most unique feature — an AI that generates a complete UI layout:

1. User types a natural language intent: `"Track my gym sessions this week"`
2. Request streams to `/genui/compose` via SSE
3. Backend AI returns a JSON layout spec: `{ nodes: [{ type: "HabitGrid", props: {...} }, { type: "FocusTimer", props: {...} }] }`
4. `GenUIRenderer` maps each node type to a catalog component and renders them
5. The canvas can be pinned (saved) and reloaded

### Widget Catalog
Each widget in `src/components/catalog/` is a standalone React component:

| Widget | Description |
|---|---|
| `TaskCard` | Displays a single task in `row` (full-width list) or `card` (compact kanban) variant. Shows priority color bar, due date, tags, overdue badge, estimated time, and action buttons |
| `KanbanBoard` | Three columns (Todo / In Progress / Done). Tasks draggable between columns via status update |
| `CalendarView` | Full month grid. Days with tasks show colored priority dots. Click to filter |
| `HabitGrid` | 7×N grid like GitHub contribution graph. Tracks binary daily completion per habit |
| `FocusTimer` | Pomodoro timer (25/5 min) with SVG ring progress, start/pause/reset |
| `BudgetTracker` | Income vs expense tracker with running balance |
| `ProgressDashboard` | Recharts bar chart showing XP earned per day |
| `ChecklistGroup` | Named groups of checkbox items |
| `CountdownRing` | SVG ring that counts down to a deadline |
| `Timeline` | Chronological list with time markers |

### Gamification
- **XP system** — completing tasks awards XP based on priority (`urgent`=100, `high`=60, `medium`=30, `low`=10)
- **Levels** — every 500 XP advances the level. Level displayed in sidebar and dashboard
- **Streaks** — backend tracks consecutive login days. Displayed in dashboard stats
- **Confetti** — `canvas-confetti` fires on task completion via `useXpReward` hook

## Authentication

Authentication uses **NextAuth v5** with two providers:

### Credentials (email + password)
```ts
// src/auth.ts
CredentialsProvider({
  authorize: async (credentials) => {
    const res = await fetch(`${API_URL}/auth/login`, {
      method: 'POST',
      body: JSON.stringify({ email, password }),
    })
    const data = await res.json()
    if (!res.ok) return null
    return { ...data.user, backendToken: data.token }
  }
})
```

The backend JWT token is stored inside the NextAuth session object (encrypted, stored in an httpOnly cookie by NextAuth). It is **never written to localStorage** — this prevents XSS theft of tokens.

### Google OAuth
```ts
GoogleProvider({
  clientId: process.env.GOOGLE_CLIENT_ID,
  clientSecret: process.env.GOOGLE_CLIENT_SECRET,
})
```

On Google sign-in, the frontend receives the Google ID token, sends it to `/auth/google`, and the backend verifies it with Google's OAuth2Client, creates or retrieves the user, and returns a JWT.

### Session Token Flow
```
User logs in → NextAuth creates encrypted session cookie
→ API calls read session via getSession()
→ backendToken extracted from session
→ Passed as Authorization: Bearer <token> header
→ Backend validates JWT
```

The `getSessionToken()` function in `src/lib/api.ts` uses promise deduplication to prevent multiple concurrent `/api/auth/session` fetches:

```ts
let _sessionTokenPromise: Promise<string | null> | null = null

async function getSessionToken() {
  if (_sessionToken) return _sessionToken
  if (_sessionTokenPromise) return _sessionTokenPromise  // reuse in-flight request
  _sessionTokenPromise = fetchSession().finally(() => {
    _sessionTokenPromise = null
  })
  return _sessionTokenPromise
}
```

## State Management

Zustand is used for all client state. Two stores:

### `useTaskStore`
```ts
interface TaskStore {
  tasks: Task[]
  lists: TaskList[]
  loading: boolean
  error: string | null
  fetchTasks(filters?) → void
  fetchLists() → void
  createTask(data) → Task
  updateTask(id, data) → void      // optimistic update with rollback
  completeTask(id) → { xpAwarded }
  deleteTask(id) → void            // optimistic delete with rollback
  createList(data) → void
}
```

**Optimistic updates** work like this — for `deleteTask`:
```ts
deleteTask: async (id) => {
  const previous = get().tasks           // snapshot current state
  set(s => ({ tasks: s.tasks.filter(t => t._id !== id) }))  // update instantly
  try {
    await tasksApi.delete(id)            // confirm with server
  } catch {
    set({ tasks: previous })             // restore on failure
  }
}
```

### `useUserStore`
Holds the user profile (name, email, image, xp, level, streak). Updated after login and after task completion (XP changes).

## API Layer

All API calls are centralised in `src/lib/api.ts`. An Axios instance is created with the backend base URL and an interceptor that attaches the JWT:

```ts
api.interceptors.request.use(async (config) => {
  const token = await getSessionToken()
  if (token) config.headers.Authorization = `Bearer ${token}`
  return config
})
```

API modules:
- `tasksApi` — `getAll`, `getOne`, `create`, `update`, `complete`, `delete`, `suggest`
- `listsApi` — `getAll`, `create`, `update`, `delete`
- `usersApi` — `getMe`, `updateMe`, `updateNotifications`
- `genUIApi` — `compose` (SSE stream), `clarify`, `getCanvases`, `saveCanvas`, `togglePin`
- `mailApi` — `sendPlan`

## Generative UI System

The GenUI system is what makes TaskVerse unique. Here is the full data flow:

```
User types intent
      ↓
POST /genui/compose  (with intent + existing lists context)
      ↓
Backend AI (Gemini primary, OpenRouter fallback, Groq fallback)
generates layout JSON:
{
  "nodes": [
    { "type": "KanbanBoard", "props": { "title": "Workout Tracker" } },
    { "type": "HabitGrid", "props": { "habits": ["Gym", "Run", "Swim"] } }
  ]
}
      ↓
Streamed back via SSE (text/event-stream)
      ↓
GenUICanvas.tsx accumulates chunks → parses JSON when complete
      ↓
GenUIRenderer.tsx maps node.type → catalog component
      ↓
Component rendered with node.props
```

The `catalog-registry.ts` maps string names to components:
```ts
const REGISTRY: Record<string, React.ComponentType<any>> = {
  KanbanBoard,
  HabitGrid,
  FocusTimer,
  BudgetTracker,
  CalendarView,
  // ...
}
```

## Security

### HTTP Security Headers (`next.config.ts`)

Every response from the Next.js server includes:

| Header | Value | Purpose |
|---|---|---|
| `Content-Security-Policy` | `default-src 'self'` + allowlisted sources | Prevents XSS by blocking unauthorised scripts/styles |
| `Strict-Transport-Security` | `max-age=63072000; includeSubDomains` | Forces HTTPS for 2 years |
| `X-Frame-Options` | `DENY` | Prevents clickjacking in iframes |
| `X-Content-Type-Options` | `nosniff` | Prevents MIME-type sniffing |
| `Referrer-Policy` | `strict-origin-when-cross-origin` | Controls referrer header leakage |
| `Permissions-Policy` | `camera=(), microphone=(), geolocation=()` | Disables unused browser APIs |

### Authentication Security
- JWT stored in NextAuth **httpOnly cookie** — JavaScript cannot read it, preventing XSS token theft
- Google ID tokens verified server-side with `google-auth-library`, not trusted from client
- Session token promise deduplication prevents race conditions in concurrent API calls

### Input Validation
- All forms use **React Hook Form + Zod** schemas — invalid data never reaches the API
- Password field: minimum 8 characters enforced on client

### What Was Fixed During Security Review
- Removed direct `localStorage.setItem('token', ...)` call in `AppInit.tsx`
- Added `{ stream: true }` to `TextDecoder` for correct multi-byte UTF-8 handling in SSE
- Added non-2xx handling in SSE reader (was silently continuing on server errors)
- Added optimistic rollback to all mutating store actions

## SEO

### Technical SEO
- **Dynamic sitemap** at `/sitemap.xml` — lists public pages with `lastModified` and `priority`. Auth pages (`/login`, `/register`) excluded — they are `noindex` and including them in the sitemap sends contradictory signals to crawlers
- **robots.txt** at `/robots.txt` — disallows `/app/*` (private app routes) and `/api/*`
- **Canonical URLs** on every page prevent duplicate content indexing
- **Web App Manifest** at `/manifest.webmanifest` for PWA support

### Structured Data
Root layout includes JSON-LD `WebApplication` schema:
```json
{
  "@context": "https://schema.org",
  "@type": "WebApplication",
  "name": "TaskVerse",
  "description": "AI-powered todo app that builds itself",
  "applicationCategory": "ProductivityApplication",
  "operatingSystem": "Web"
}
```

### On-Page SEO
- Single clear `H1` per page
- Title tags: `TaskVerse — AI Todo That Builds Itself | Manage Tasks with AI` (~55 chars)
- Meta descriptions: ~155 chars, unique per page, describe actual content
- `preconnect` to Google Fonts to reduce render-blocking

### GSAP Animation Fix
The landing page uses GSAP for scroll animations. Originally headings started at `opacity: 0` — meaning crawlers (which don't run JavaScript animations) would index the page as having invisible headings, killing keyword signal.

**Fix:** Changed from `gsap.from(el, { opacity: 0 })` to `gsap.fromTo(el, { y: 30 }, { y: 0 })` — headings are always visible in the initial HTML, only their vertical position animates.

## Environment Variables

Copy `.env.example` to `.env.local` and fill in:

```env
NEXT_PUBLIC_API_URL=http://localhost:3001        # Backend URL
NEXTAUTH_URL=http://localhost:3000               # This app's URL
NEXTAUTH_SECRET=                                 # Random 32-byte base64 string
GOOGLE_CLIENT_ID=                                # Google Cloud Console OAuth client
GOOGLE_CLIENT_SECRET=                            # Google Cloud Console OAuth secret
```

Generate `NEXTAUTH_SECRET`:
```bash
node -e "console.log(require('crypto').randomBytes(32).toString('base64'))"
```
